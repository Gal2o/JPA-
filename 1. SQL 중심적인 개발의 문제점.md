- ## SQL 중심적인 개발의 문제점
  - #### 요즘 시대는 객체를 RDBMS에 관리한다.
  - #### 무한 반복, 지루한 코드
    - #### CRUD <br><br> INSERT INTO <br><br> UPDATE <br><br> SELECT <br><br> DELETE <br><br> 자바 객체 ↔ SQL 객체 매핑
  - #### SQL에 의존적인 개발을 피하기 어렵다.
  - #### 현실적으로 RDBMS가 가장 많이 쓰이기 때문에 다른 저장소가 대안할 수 없다.
  - ### 객체와 RDBMS의 차이
    - ### 상속 <br><br> ![image](https://user-images.githubusercontent.com/35948339/143820031-5e119262-f8fb-4c55-870f-14ebed241eda.png)
      - #### 객체에는 `상속 관계가 있지만`, RDBMS는 `상속 관계가 없다.`
      - #### RDBMS에서 `Album을 저장하는 과정`을 보면
        - #### `객체를 분해`하고, `INSERT INTO ITEM ITEM_ID`, `INSERT INTO ALBUM ITEM_ID, ARTIST...`
      - #### 저장한 `Album 테이블을 조회`할 땐
        - #### 각각의 테이블에 따른 `Join SQL 작성`
        - #### 각각의 `객체 생성해서 조회` ▶ 모든 테이블 반복 `(복잡하다)`
        - #### `DB에 저장할 객체`에는 `상속 관계를 쓰지 않는다.`
    ------
    - ### 연관관계 <br><br> ![image](https://user-images.githubusercontent.com/35948339/143820770-19e72ecd-f07c-435b-8950-193d23b29d11.png)
      - #### 객체에는 `참조를 통해 가져오고 (member.getTeam())` <br><br> RDBMS는 `외래키`를 통해 `Join on m.team_id = t.team_id`
      ``` java
      
        // member만 조회
        memberDAO.getMember()
        
        // Member와 Team조회
        memberDAO.getMemberWithTeam();
      ```
